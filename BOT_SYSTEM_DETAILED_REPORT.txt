4) DETAILED FINDINGS (PRIORITIZED)
============================================================

[HIGH] Whitelist admin APIs are account-capability scoped, not guild-capability scoped
- Evidence: routes/apiRoutes.js lines ~133-143
  - /api/whitelist/list, /approve/:id, /reject/:id, /config/:guildId use requireAccountCapability("manage_settings")
  - No requireGuildCapability checks on notificationGuildId/guildId route param
- Impact:
  - Admin with manage_settings capability could potentially act on guild IDs outside intended scope.
- Recommendation:
  1. Add requireGuildCapability("manage_settings") for config endpoints by params guildId.
  2. For approve/reject/resend endpoints, validate notificationGuildId against req.session.guildAccess server-side before action.
  3. Validate channel ownership against validated guild only.

[HIGH] rejectionTemplate is not persisted by whitelist config service
- Evidence:
  - controllers/whitelistController.js passes rejectionTemplate in configData.
  - services/whitelistConfigService.js destructures only { channelId, embedTemplate, roleId } and stores only embedTemplate.
- Impact:
  - Rejection custom template appears editable in UI but not reliably stored.
- Recommendation:
  - Extend saveWhitelistConfig to include rejectionTemplate in findOneAndUpdate payload.

[MEDIUM] Rejection notification builder uses approval defaults when template fields missing
- Evidence: services/whitelistNotificationService.js customizeEmbedTemplate() defaults to approval text/title/color for all calls.
- Impact:
  - Rejection alerts may show wrong fallback wording/style if rejection template is partial.
- Recommendation:
  - Split template customizers by event type or pass explicit default set for approve/reject.

[MEDIUM] Public whitelist frontend hardcodes backend API endpoint
- Evidence: dashboard/whitelist/whitelist.js line 7 uses fixed Railway URL.
- Impact:
  - Reduced portability across staging/prod/custom domains.
- Recommendation:
  - Use relative /api/whitelist/apply when hosted with dashboard, with optional override via global config variable.

[LOW] No automated tests (unit/integration/e2e)
- Evidence: package.json has start/dev only.
- Impact:
  - High regression risk across rapidly evolving feature set.
- Recommendation:
  - Add baseline test harness for services and API contract smoke tests.

[LOW] CORS origin list is static
- Evidence: server/app.js lines ~37-45.
- Impact:
  - Environment-specific deploys need code change for new origins.
- Recommendation:
  - Move allowed origins to env-driven comma-separated config.

5) OPERATIONAL BEHAVIOR VALIDATION
============================================================
Observed design behavior from code:

- Bot startup:
  - DB connect -> app listen -> Discord login -> command register -> reminder start
  - Scheduler + Minecraft monitor start in setImmediate lifecycle hook.

- Scheduler:
  - Recurring schedules registered via cron.
  - One-time schedules discovered each minute via DB query.
  - Duplicate run prevention for same schedule ID while executing.

- Minecraft monitor:
  - Poll interval: 20s.
  - Startup burst: 3 checks every 5s after start.
  - Alerts only on state transition (not continuous spam in steady state).
  - Offline transition triggers auto-restart attempt with cooldown.

- Command system:
  - Commands loaded from bot/commands recursively.
  - Optional guild-scoped registration with COMMAND_GUILD_ID.
  - Execution guarded by commandAccessService + metrics.
